Introduction

Before the current revision, this web site was most recently written in Angular 2. That was in April 2016, when Angular 2 was still in beta. I created it quite manually - not even knowing the Angular 2 CLI existed. I have rewritten it now that Angular 2 is RTM, using Angular CLI Beta 15. The site now also uses .NET Core, and it's all comprised in a Visual Studio 2015 solution. It's not easy to get all these things working together, so he's my first attempt at an actual blog to share some knowledge.

The entire solution is available on GitHub.

Step 4: Fixing up Visual Studio

New we can build in Visual Studio - and get a big fat error.

Error CS0234 The type or namespace name 'Web' does not exist in the namespace 'System' (are you missing an assembly reference?)

This is because the Angular 2 CLI installed the selenium-webdriver node module, and Visual Studio is trying to build everything it finds in node_modules.

Let's tell VS to ignore node_modules. Modify the project json...

"buildOptions": {{ '{' }}
    "emitEntryPoint": true,
    "preserveCompilationContext": true,
    "compile": {{ '{' }}
        "exclude": [ "node_modules" ]
    }
}

Now the Visual Studio solution builds with no errors.

Step 5: Building the Angular 2 CLI Project to wwwroot

By default, the Angular 2 CLI builds to /dist. Visual Studio needs this to be /wwwroot instead.

Angular 2 CLI has a parameter in it's angular-cli.json called 'outDir', but changing this currently does not work.

We can compensate for this on the command line when we build. "-o wwwroot" ensures ng builds to right location for Visual Studio.

$ ng build -o wwwroot

And now that were are building to wwwroot, git will start seeing all those files - that we do not need.

Step 8: Make Visual Studio Run the Project

At this point, if we run the project from Visual Studio, it's broken. It still thinks it's a WebApi project only, and we deleted it's only controller. It's quite sad really.

So let's fix this up. First, in the project.json file, we add a new dependency - Microsoft.AspNetCore.StaticFiles.

And in startup.cs, we enable the serving of static files.

public void Configure(IApplicationBuilder app, IHostingEnvironment env, ILoggerFactory loggerFactory)
{{ '{' }}
    loggerFactory.AddConsole(Configuration.GetSection("Logging"));
    loggerFactory.AddDebug();
    app.UseMvc();
    app.UseDefaultFiles();
    app.UseStaticFiles();
}

And finally, right click the web project in Visual Studio... Properties... Debug. Keep the Launch URL option checked, but clear the value.

Now we can build the Visual Studio solution and the Angular 2 CLI project is working.

See the backslash? That needs to be fixed.

The Angular 2 CLI also added the NewsComponent to the declarations in our App module (app.module.ts). It did not, however, and the DataService to the providers section. We want the access the DataService from anywhere, so we add that as well.

Conclusion

I like generating all the Angular files on the command line - that saves lots of time. I also like that it also generates all the spec files for writing tests - it is something that is often overlooked in javascript/Typescript projects and this little head start will help a lot of people. I think Angular 2 on the client paired with .NET on the server for data access is a pretty nice way to go, and having it all combined in one solution is convenient.

I somewhat like the CLI build process but occasionally things do not work right and it can take a while to figure out what the cause it. Is it the CLI? Is it something with webpack? Is it something with Angular 2 itself? It can be a challenge to figure out the issue when things go wrong. Also, the webpack config is not exposed in anyway - you may want to do something that webpack allows - but the CLI won't let you. Their reasoning here is that they want the ability to switch to something else besides webpack in the future, and not break your code.

I'm still not completely sold on the Angular 2 CLI. One thing it eliminates is the ability to dynamically generate templates on the server using the templateUrl property of a component. This is because the build accesses the templateUrl, and stores the response as an inline template - if the templateUrl is dynamic it throws an error. I pointed this out and it will not be changed. I supposed you could use the CLI to generate the files but NOT use the build process.
